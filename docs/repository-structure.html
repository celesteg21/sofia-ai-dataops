Guia tecnica del repositorio

Sofia AI DataOps

Plataforma inicial para analizar incidentes de Airflow con agentes de IA, una API FastAPI, LangGraph para orquestacion, Qdrant para recuperacion semantica y PostgreSQL para persistencia operacional.

Objetivo

Diagnosticar

Incidentes de Airflow

Recibir logs, metadata de DAGs y datos de ejecucion para generar una hipotesis de causa raiz, severidad y acciones recomendadas.

Orquestar

Agentes con LangGraph

Modelar el analisis como un grafo extensible: recuperar contexto, clasificar el problema y recomendar pasos de remediacion.

Operar

Base DataOps

Separar API, servicios, agentes, persistencia y observabilidad para que el sistema pueda crecer con evaluaciones, trazas, memoria y revisiones humanas.

Principio de diseno

Capas con responsabilidades claras

La API no contiene logica de diagnostico; los servicios orquestan casos de uso; los agentes encapsulan el razonamiento; las bases de datos viven detras de clientes propios. Esta separacion hace mas facil testear, reemplazar modelos, agregar memoria semantica o cambiar infraestructura sin reescribir toda la aplicacion.

  • FastAPI expone contratos HTTP.
  • LangGraph define el flujo de agente.
  • Qdrant queda preparado para incidentes similares.
  • PostgreSQL guarda los analisis generados.

Que hace cada parte

src/sofia_ai_dataops/api
Crea la aplicacion FastAPI, registra rutas y conecta dependencias. Incluye /health y /api/v1/incidents/analyze.
src/sofia_ai_dataops/agents
Contiene el grafo de LangGraph, el estado compartido del agente y los nodos de recuperacion, clasificacion y recomendacion.
src/sofia_ai_dataops/services
Orquesta el caso de uso principal: recibe un incidente, ejecuta el grafo, arma la respuesta, persiste el resultado y emite eventos de observabilidad.
src/sofia_ai_dataops/schemas
Define contratos Pydantic para requests y responses. Es la frontera tipada entre el cliente HTTP y el dominio interno.
src/sofia_ai_dataops/db
Encapsula la persistencia. postgres.py guarda analisis en PostgreSQL y qdrant.py prepara la busqueda de incidentes similares.
src/sofia_ai_dataops/core
Centraliza configuracion con variables de entorno y logging estructurado con structlog.
src/sofia_ai_dataops/ingestion
Punto de entrada para normalizar logs y eventos de Airflow antes de indexarlos o analizarlos.
src/sofia_ai_dataops/observability
Hooks para eventos de aplicacion. Hoy registra incidentes analizados; luego puede crecer hacia trazas, metricas y auditoria.
tests
Pruebas unitarias iniciales para endpoints y nodos del agente. Es la base para agregar evaluaciones de prompts, regresiones de incidentes y pruebas de integracion.
docker-compose.yml
Levanta API, PostgreSQL y Qdrant para desarrollo local con una sola orden.

Flujo de Analisis

1. Request El cliente envia DAG, task, run_id, logs y metadata.
2. Servicio IncidentAnalysisService prepara el estado del grafo.
3. Recuperacion Qdrant queda como frontera para traer incidentes similares.
4. Agente LangGraph clasifica severidad y genera recomendaciones.
5. Persistencia PostgreSQL guarda el analisis y la API devuelve la respuesta.

Archivos Clave

pyproject.toml

Declara dependencias, version de Python, paquetes, configuracion de pytest, Ruff y mypy.

.env.example

Lista variables necesarias para API, PostgreSQL, Qdrant y modelos de IA.

Makefile

Atajos para instalar, ejecutar, testear, linting y levantar Docker.

Dockerfile

Imagen de la API Python con instalacion editable del paquete.

docker/postgres-init.sql

Crea la tabla inicial incident_analyses para guardar diagnosticos.

.github/workflows/ci.yml

Pipeline inicial para instalar dependencias, correr Ruff, mypy y pytest.
