V0.1 — Incident Analyzer

Sofia entiende por qué
falló tu pipeline

Una IA que analiza incidentes de datos, reúne contexto distribuido y devuelve un diagnóstico estructurado con causa raíz, impacto y plan de acción.

POST /incidents/analyze
El problema

Los incidentes de datos son costosos de investigar

Cuando un pipeline falla a las 3am, la información para entender qué pasó está dispersa en múltiples sistemas. Un Data Engineer tiene que navegar todo eso manualmente, bajo presión, antes de poder actuar.

📋

¿Qué pipeline falló?

Y cuál fue exactamente el error. No siempre está claro desde la alerta.

📜

¿Qué dice el log?

Los logs pueden tener cientos de líneas. Encontrar la línea relevante toma tiempo.

🗄️

¿Falta una partición?

¿Qué dataset o tabla upstream no tiene los datos esperados para hoy?

📊

¿Qué reportes afecta?

¿Quién está viendo datos desactualizados o rotos en este momento?

📚

¿Existe un runbook?

¿Alguien ya resolvió esto antes? ¿Cuál era el procedimiento correcto?

🔁

¿Cómo evitarlo?

Una vez resuelto, ¿qué cambio estructural evitaría que pase de nuevo?

💡

Sofia automatiza este proceso. Recibe el ID de un incidente, reúne todo el contexto disponible (logs, metadata, runbooks, calidad de datos) y devuelve un análisis completo en milisegundos.

Cómo funciona

El flujo de análisis de un incidente

Cuando llegás el endpoint con un incident_id, Sofia ejecuta cinco pasos en secuencia:

1
Carga el incidente
Lee el JSON del incidente desde mock_environment/incidents/
→
2
Reúne contexto
Logs, metadata del pipeline, runbooks relevantes, resultados de calidad
→
3
Consulta al LLM
Pasa todo el contexto al servicio LLM (mock o real)
→
4
Sintetiza
El LLM construye causa raíz, evidencia, impacto y recomendaciones
→
5
Responde
Devuelve el análisis completo como JSON estructurado

El endpoint

POST http://localhost:8000/incidents/analyze
Content-Type: application/json

{
  "incident_id": "inc_001"
}

Caso de uso concreto

Incidente inc_001 — Reporte de Revenue caído

El pipeline daily_revenue falla todos los días a las 3am cuando la ingesta se retrasa. Sofia analiza el incidente y entrega el diagnóstico completo.

1

Se recibe el incidente

Sofia carga incidents/inc_001.json y extrae el contexto inicial:

{
  "incident_id": "inc_001",
  "pipeline":     "daily_revenue",
  "status":       "failed",
  "failed_at":    "2026-06-05T03:14:22Z",
  "error_message":"PartitionNotFoundError: Partition dt=2026-06-04 does not exist in table raw.transactions",
  "depends_on":   ["ingestion_transactions"],
  "severity":     "high"
}
2

Lee los logs del pipeline

LogReader carga logs/daily_revenue.log y extrae las líneas con errores:

2026-06-05 03:10:00 INFO  [daily_revenue] Pipeline started
2026-06-05 03:10:01 INFO  Checking upstream dependencies...
2026-06-05 03:10:02 INFO  Querying raw.transactions for partition dt=2026-06-04
2026-06-05 03:10:03 ERROR PartitionNotFoundError: Partition dt=2026-06-04 does not exist
2026-06-05 03:10:03 ERROR Pipeline failed after 3s. Exit code: 1
2026-06-05 03:10:04 INFO  Sending alert to #data-platform-oncall
3

Consulta metadata del pipeline

MetadataReader carga metadata/daily_revenue.json para entender dependencias y downstream:

{
  "depends_on_pipelines": ["ingestion_transactions"],
  "downstream_reports":   ["Revenue Dashboard", "CFO Daily Brief"],
  "expected_freshness_hours": 2,
  "sla_minutes": 60
}
4

Encuentra el runbook relevante

RunbookReader detecta PartitionNotFoundError en el error y carga runbooks/missing_partition.json:

{
  "immediate_action": "1. Check status of ingestion_transactions...\n2. Rerun ingestion for dt=2026-06-04...\n3. Validate partition exists...\n4. Rerun daily_revenue.",
  "prevention": "Add a freshness check sensor before running dependent report pipelines."
}
5

Verifica el resultado de calidad

El ContextService carga quality_results/daily_revenue.json que confirma el problema de freshness:

{
  "check_type":      "freshness",
  "status":          "failed",
  "detail":          "Table raw.transactions last updated 2026-06-03T22:00:00Z. Expected freshness: 2h. Actual lag: 29h.",
  "threshold_hours": 2,
  "actual_lag_hours":29
}
6

El LLM sintetiza y responde

El MockLLMService combina todo el contexto y genera el análisis final. Sin modelo externo, sin API key.

Output

El análisis completo que devuelve Sofia

Siete campos estructurados, accionables. Cada uno derivado del contexto real del incidente. 

{
  "incident_id":          "inc_001",

  "summary":               "Pipeline 'daily_revenue' failed. Error: PartitionNotFoundError:
                             Partition dt=2026-06-04 does not exist in table raw.transactions",

  "probable_root_cause":   "Table raw.transactions last updated 2026-06-03T22:00:00Z.
                             Expected freshness: 2h. Actual lag: 29h.",

  "evidence":              [
    "ERROR [daily_revenue] PartitionNotFoundError: Partition dt=2026-06-04 does not exist",
    "ERROR [daily_revenue] Pipeline failed after 3s. Exit code: 1",
    "Data quality: Table raw.transactions last updated 2026-06-03T22:00:00Z. Actual lag: 29h."
  ],

  "business_impact":       "Reports at risk: Revenue Dashboard, CFO Daily Brief.
                             Quality check: Table raw.transactions last updated 2026-06-03...",

  "recommended_action":    "1. Check status of ingestion_transactions for the missing date.
                             2. If failed: rerun ingestion_transactions for dt=2026-06-04.
                             3. Validate: SELECT COUNT(*) FROM raw.transactions WHERE dt='2026-06-04'.
                             4. Rerun daily_revenue pipeline once partition is confirmed.",

  "long_term_improvement": "Add a freshness check sensor (e.g. PartitionSensor or
                             ExternalTaskSensor) before running dependent report pipelines.",

  "confidence":            "medium"
}
summary

Qué pipeline falló y cuál fue el error exacto.

Pipeline 'daily_revenue' failed. Error: PartitionNotFoundError…

probable_root_cause

La causa raíz derivada del contexto disponible, no solo del mensaje de error.

Table raw.transactions last updated 29h ago. Expected freshness: 2h.

evidence

Lista de líneas de log y resultados de calidad que sustentan el diagnóstico.

["ERROR PartitionNotFoundError…", "Data quality: lag 29h"]

business_impact

Qué reportes o datasets downstream están afectados en este momento.

Reports at risk: Revenue Dashboard, CFO Daily Brief.

recommended_action

Pasos concretos y ordenados del runbook para resolver el incidente ahora.

1. Check ingestion_transactions… 2. Rerun… 3. Validate… 4. Rerun report.

long_term_improvement

Cambio estructural para evitar que este incidente vuelva a ocurrir.

Add a freshness check sensor before running dependent pipelines.

LLM como plug

Sin modelo hoy, con modelo mañana

LLMService es una clase abstracta. El backend se elige con una variable de entorno. Cambiar de mock a Ollama no toca ninguna lógica de negocio.

● Mock LLM activo hoy

  • No requiere modelo instalado
  • Respuesta determinista (mismos inputs → mismo output)
  • Deriva el análisis del contexto estructurado (runbook, metadata, quality)
  • Tests pasan sin dependencias externas
  • Activar: LLM_BACKEND=mock

● Ollama LLM stub listo para V0.2

  • Requiere Ollama corriendo localmente
  • Modelo configurable: llama3, mistral, phi3, etc.
  • Genera análisis en lenguaje natural real
  • Sin cloud, sin API key externa
  • Activar: LLM_BACKEND=ollama
🔌

El diseño es framework-agnostic. Ninguna capa de negocio conoce a LangGraph, CrewAI ni Strands. Cuando se necesiten, se agregan como implementaciones de las interfaces existentes — sin reescribir lógica central.

Qué sigue

Roadmap de Sofia

Cada versión agrega valor independiente sin romper lo anterior.

V0.1 Incident Analyzer Completo

Diagnóstico estructurado a partir de contexto local. Mock LLM, arquitectura framework-agnostic, OrchestratorAdapter como interfaz.

V0.2 Runbook RAG Próximo

Búsqueda semántica sobre runbooks con embeddings locales vía Ollama. Reemplaza el keyword matching actual — Sofia encuentra el runbook correcto aunque el error use otras palabras.

V0.3 Metadata & Lineage Awareness Pendiente

Parseo de grafos de linaje. Impacto upstream/downstream automático. Sofia sabe qué tabla alimenta qué reporte sin que le digan explícitamente.

V0.4 Orchestrator Adapters Pendiente

Adapters reales para Airflow, Dagster, Prefect y Step Functions. Todos implementan la interfaz OrchestratorAdapter ya definida en V0.1.

V0.5 Recommendation Engine Pendiente

Detección de patrones en incidentes recurrentes. Sofia identifica que el mismo pipeline falla todos los lunes y propone mejoras estructurales priorizadas.

V0.6 Human Approval Actions Pendiente

Sofia propone acciones concretas (rerun, trigger, alert). El ingeniero aprueba antes de que se ejecuten. Audit trail completo.

V1.0 Multi-orchestrator DataOps Copilot Pendiente

Vista unificada de todos los orquestadores. LLM real (Ollama / cloud). Integración con Slack y PagerDuty. Frameworks de agentes como LangGraph o Strands como implementaciones pluggables.
